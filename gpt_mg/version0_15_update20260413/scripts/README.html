DET

DET를 어떻게 읽어야 하나

DET는 단순 exact match가 아니라, service 해석·의미 정렬·GT coverage·값 흐름을 여러 축으로 합산한 점수입니다.

프로파일장점약점권장 용도
legacy기존 실험과 호환, 빠르고 비교적 관대함값 흐름, receiver coverage, numeric/enum grounding을 후하게 줄 수 있음historical comparison, GA search
strictGT service coverage, receiver coverage, dataflow, numeric/enum grounding을 엄격히 반영service equivalence가 없는 케이스는 다소 보수적으로 볼 수 있음최종 품질 검증, 리포트, 논문용 해석
권장 해석: historical/GA 비교는 legacy, 실제 품질 리포트와 acceptance는 strict를 우선으로 보세요.

det_score

최종 DET 점수. 0~100.

det_gt_exact

GT와 exact match 여부.

det_gt_similarity

GT와의 전체 유사도. exact가 아니어도 구조가 비슷하면 일부 점수를 받습니다.

det_service_match

생성 코드의 서비스가 schema에 얼마나 잘 resolve되는지.

det_arg_type_ok

각 서비스의 인자 타입과 개수가 schema expectation과 얼마나 맞는지.

det_precondition_ok

전제조건이나 guard가 필요한 경우 그것이 반영됐는지.

det_semantic_ok

명령 의미와 action 방향이 얼마나 맞는지.

det_min_extraneous

명령에 없는 불필요한 action이 섞였는지.

det_gt_service_coverage

GT에 필요한 서비스를 생성 코드가 모두 포함했는지.

det_gt_receiver_coverage

GT의 receiver scope와 instance coverage를 얼마나 잘 덮었는지.

det_dataflow_score

sensor/value를 읽은 뒤 실제 downstream 함수에 전달했는지.

det_numeric_grounding

시간, 채널, 온도 같은 숫자 정보가 정확히 반영됐는지.

det_enum_grounding

enum 값(dry, emergency 등)이 정확한지.

FAILURE-REASONS

failure_reasons 상세 해설

failure_reasons는 런타임 크래시 로그가 아니라, 왜 감점됐는지를 설명하는 진단 태그입니다.

failure_reason무슨 뜻인가대표 원인읽는 팁
invalid_json출력이 JSON으로 파싱되지 않음자유 텍스트, braces mismatch, explanation 혼입맨 앞단 실패. 나머지 지표보다 먼저 해결
schema_missing_keysJSON은 맞지만 name / cron / period / code 중 필수 키 누락code 없이 script만 반환출력 스키마 프롬프트 확인
no_parseable_member_accesscode는 있지만 (#Device).Service(...) 형식으로 parse 불가JOILang 대신 의사코드 반환코드 포맷을 먼저 고정
unknown_service:<member>schema에서 매핑 불가한 service/memberinvented helper, 심한 오타, 잘못된 camel/snake caseschema grounding 강화
service_match전체 호출 중 일부가 schema에 정확히 resolve되지 않음맞는 서비스와 틀린 서비스 혼재unknown service와 함께 자주 나타남
arg_type서비스는 맞지만 인자 타입/개수/순서 불일치숫자 대신 string, enum 대신 자유 텍스트numeric/enum grounding과 같이 해석
precondition필요한 전제조건/guard가 부족power/switch guard 없이 바로 action명령 의도보다 실행 안전성 문제
semantic명령 의미와 action 방향이 약하게 맞음increase/decrease, lock/unlock 반대service 이름은 맞아도 의미가 어긋날 수 있음
extraneous명령에 없는 불필요한 action 추가speaker가 필요 없는데 speaker까지 사용과생성 신호
gt_mismatchGT와 exact match는 아님매우 넓은 요약 태그이것만 보면 부족. 아래 coverage 항목을 같이 봐야 함
gt_service_coverageGT에 필요한 서비스 일부 누락HumiditySensor + Speaker 중 Speaker 누락가장 자주 중요한 감점 사유 중 하나
gt_receiver_coverageGT의 receiver scope 일부 누락Front/Back 둘 다 필요한데 하나만 처리all(...) 계열 명령에서 중요
dataflow읽은 값이 실제 sink로 전달되지 않음sensor value를 읽기만 하고 speak/set에 안 씀strict에서 핵심
numeric_grounding숫자/단위 정보가 정확하지 않음30분을 1800초 대신 ms 표현 또는 잘못된 수치strict에서 강하게 감점
enum_groundingenum 값이 틀림dry 대신 warmstrict에서 강하게 감점
가장 흔한 오해: gt_mismatch 하나만 보고 판단하면 원인을 놓치기 쉽습니다. gt_service_coverage, gt_receiver_coverage, dataflow, numeric_grounding, enum_grounding을 같이 봐야 합니다.

PLAYBOOK

문제 분석 순서

row report와 CSV를 볼 때 어디서부터 읽으면 빠른지, 실무 기준 디버깅 순서를 정리했습니다.

row report 먼저 볼 때

  • DET와 failure_reasons를 먼저 확인
  • gt_service_coverage, gt_receiver_coverage, dataflow를 바로 같이 보기
  • GT vs Generated JOICode를 좌우 배치로 비교
  • 필요하면 prompt_tokens, llm_latency_sec, peak_vram_gb까지 확인

CSV / audit으로 깊게 볼 때

  • failure_reason_summary.csv에서 모델별 failure distribution 확인
  • row_comparison.csv에서 낮은 DET row 필터링
  • audit_det.py로 legacy vs strict 차이 확인
  • compare_benchmarks.py로 baseline vs retrieval 변화 확인
실무 팁: legacy high && strict low인 row는 evaluator가 후하게 준 의심 케이스라서 우선순위 높게 보는 것이 좋습니다.
