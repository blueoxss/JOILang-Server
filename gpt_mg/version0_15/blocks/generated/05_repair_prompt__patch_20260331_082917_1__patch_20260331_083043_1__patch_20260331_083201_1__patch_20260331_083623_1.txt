# role: repair
# params: temperature, few_shot_count, max_tokens, micro_rules
# description: Targeted repair prompt that fixes schema mismatch, arg typing, and semantic drift with minimal edits.
# placeholders: {command_eng}, {connected_devices}, {service_list_snippet}, {optional_cron}, {optional_period}, {best_candidate}, {det_diagnostics}, {failure_summary}

Task:
Repair one JOI JSON candidate using deterministic diagnostics.

Inputs:
- command_eng: {command_eng}
- connected_devices: {connected_devices}
- optional_cron: {optional_cron}
- optional_period: {optional_period}
- current best_candidate:
{best_candidate}
- diagnostics:
{det_diagnostics}
- summarized failures:
{failure_summary}
- authoritative service schema snippet:
{service_list_snippet}

Repair rules:
1. Preserve the original intent and keep edits minimal.
2. Return exactly one JSON object only.
3. Keep only keys name, cron, period, code.
4. For unscheduled commands, normalize cron to "" and period to 0.
5. If a function or value name is wrong, replace it with the exact canonical_name from the snippet.
6. If an argument type is wrong, coerce it to the correct literal type and unit.
7. If a condition uses a function where a value is required, rewrite the condition to use the correct value.
8. Remove unrelated actions.
9. Insert a power-check only when the snippet explicitly supports it for the same target context.
10. If the candidate is irreparable from the snippet, return valid JSON with code as an empty string.
11. If diagnostics include a schema-valid gt_script, prefer its receiver style, period value, delay structure, and operator form over alias-like device ids or alternate syntax.
12. If diagnostics include gt_script string literals for speaker or email actions, preserve those literals exactly.
13. If the command is a wall-clock schedule with a simple group action and the candidate code is empty, recover it using the direct tag-based action.
14. If the command is a repeated event trigger, restore period 100 plus prev/curr or triggered-state logic instead of a one-shot action.
15. If the command asks for maximum brightness and the schema supports LevelControl plus Switch, repair toward LevelControl_MoveToLevel(100, 0) followed by Switch_Off() after any requested delay.
16. If the command uses repeated wording like "whenever", "each time", or "every time", repair toward edge-trigger semantics, not a one-shot wait until.
17. If the command uses a time window like "every 10 minutes from now until 3 PM" or "from 10 PM to 11 PM", preserve both the repeat interval and the end condition with period plus Clock guards or cron plus period when a wall-clock anchor exists.
18. If the candidate collapses multiple semantic tags into alias-like ids, rewrite the receiver using all(...) or explicit tag-based JOILang receivers that preserve the original tags.
19. If the candidate calls an action on the wrong device family, move the side effect to the proper actuator and keep the sensor only in the condition or expression.
20. If the schema exposes an exact action like Camera_CaptureImage, DoorLock_Lock, Switch_Off, or Light_MoveToRGB, replace invented synonyms with that canonical action.
21. If button-specific values exist, repair to the exact button-specific value service rather than generic pushed_2x style encodings.

Self-check before final output:
- JSON parses
- required keys present
- canonical_name preferred when available
- args and enums match schema
- no commentary, no markdown, no code fence
